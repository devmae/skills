# Harness Builder

## 핵심 개념
`Agent = Model + Harness`. 모델은 CPU이고, 하네스는 모델이 무엇을 언제 보는지를 관리하는
운영체제(OS)다. 너의 임무는 하네스(제약, 피드백 루프, 문서, 도구)를 구축하는 것이지
모델을 더 똑똑하게 만드는 것이 아니다.

## 반드시 강제해야 할 5가지 원칙
1. **컨텍스트가 지시보다 우선한다** — 추상적으로 작업을 설명하지 말고, 레포의 실제 상태
   (실제 파일 경로, 실제 심볼)를 보여줘라.
2. **계획과 실행은 분리된 단계다** — 코드를 작성하기 전에 계획을 먼저 만들고 검토한다.
3. **피드백 루프는 타협 불가다** — 모든 변경은 무언가로 검증한다(테스트/린터 = 계산적,
   또는 evaluator = 추론적). 피드백 없는 하네스는 그냥 단계만 늘어난 프롬프트다.
4. **한 번에 하나씩** — 스프린트당 정확히 하나의 기능만 구현한 뒤 커밋한다.
5. **코드베이스가 곧 문서다** — 컨벤션/결정은 별도 지식 베이스가 아니라 레포 안에 둔다.

## 호출 시: 다음 5가지 아티팩트를 구축
아래 단계를 실행한다. 생성하는 내용이 환각이 아니라 그라운딩되도록, 항상 실제 레포를 먼저
조사한다(`git ls-files`, 설정 파일 읽기).

### 1. CLAUDE.md 파일 (온보딩 문서)
- 루트 `CLAUDE.md` 하나, 그리고 주요 모듈/패키지마다 하나씩.
- 섹션: 프로젝트 컨텍스트 · 코딩 컨벤션 · 아키텍처 결정 · "여기서 일하는 방식" · 현재 진행 중인 작업.
- `assets/CLAUDE.template.md`를 사용한다. 실제로 감지된 스택/구조를 바탕으로 채운다.

### 2. JSON 기능 목록 (진행 트래커 + 테스트 스위트 통합)
- `assets/features.schema.json`를 바탕으로 `harness/features.json`을 생성한다.
- 각 항목: `id`, `description`, `verify`, `priority`, `status`("pass" | "fail"), `notes`.
- Markdown이 아니라 JSON인 이유: 실행 중 에이전트가 덮어쓸 가능성이 더 낮기 때문이다.

### 3. 세션 초기화 루틴 (매번 동일한 부팅)
- `assets/session-init.sh`를 `harness/session-init.sh`로 설치한다.
- 7단계 부팅: 작업 디렉터리 확인 → git 로그 + 진행 상황 읽기 → 우선순위가 가장 높은 실패 기능 선택 → 개발 서버 시작 → e2e/스모크 검증 실행 → 기능 하나 구현 → 서술적 메시지로 커밋 + `features.json` 업데이트.

### 4. 스프린트 계약 (코딩 전에 협상)
- `assets/sprint-contract.template.md`를 사용한다. 각 기능에 대해 무엇을 만들지 + 성공을 어떻게 검증할지 계약을 만든다. 구현 전에 검토한다. 모호한 성공 기준은 거부한다.

### 5. 구조화된 / 그라운딩된 작업 템플릿
- `assets/task-template.md`를 사용한다. 코딩 전에 그라운딩된 임팩트 맵을 만든다: 실제 파일 경로, 실제 심볼 이름, 따라야 할 기존 패턴, 구체적인 수용 기준. 존재하지 않는 API나 파일 경로를 지어내지 않는다.

## 역할 (실행자와 심판을 분리)
자기 평가는 실패한다 — 에이전트는 자기 작업을 칭찬한다. 세 역할을 분리해서 유지한다:
- **Planner** — 짧은 프롬프트를 스펙 + 기능 목록으로 바꾼다.
- **Generator** — 스프린트당 기능 하나를 구현한다.
- **Evaluator** — 실행 결과를 독립적으로 검증한다.
각 역할의 전체 프롬프트는 `references/roles.md`를 읽어라.

## Build to delete (역설)
모든 하네스 구성요소는 "모델이 아직 못하는 것"에 대한 가정을 담고 있다. 모델이 개선되면 그 가정은 만료된다. 주기적으로 `harness/audit.md`의 체크리스트를 실행한다. 기능 목록을 여전히 통과시키는 가장 단순한 하네스를 선호한다.

## 완료 시 사용자에게 출력할 것
어떤 파일을 만들었는지, 각각 무엇을 하는지, `features.json`에서 어느 기능부터 시작하면 되는지 나열한다. 사용자의 승인 없이 자율 루프를 돌리지 않는다.
